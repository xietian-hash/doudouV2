# 开发规范

开发规范是软件开发全流程中需遵守的规则手册，用于约束代码组织、接口契约、数据权限、错误处理、测试、自测和发布流程。

## 1. 关键原则（必须遵守）

1. **契约优先**：接口契约先行，OpenAPI 优先，再写实现；前后端以契约联调。
2. **后端权限兜底**：前端仅控制展示，后端必须强校验当前用户数据权限与操作权限。
3. **可观测可排障**：必须有 `traceId`，日志包含 `userId`。
4. **一致性与可维护**：统一目录、统一响应结构、统一错误码、统一校验方式。
5. **每个功能必须自测**：覆盖越权、跨用户数据访问、错误入参、幂等、并发冲突和核心功能流程。
6. **命名清晰**：字段、表名、变量、业务名词命名需无歧义，一个名称只表示一种含义。
7. **领域驱动设计**：以业务领域划分模块，避免把业务规则散落在页面或控制器中。
8. **自动化**：支持自动化部署和数据库结构自动迁移。
9. **构建门禁**：提交与部署前必须通过构建、类型检查和必要测试，不允许带构建错误进入部署。
10. **规范变更需确认**：严格按照以下技术选型与规范开发，若有改动需征询用户同意。

## 3. 仓库与工程结构

### 3.1 仓库模式

采用 Monorepo：

- `apps/miniapp`：Taro + React 微信小程序。
- `apps/api`：NestJS 后端服务。
- `packages/shared`：共享类型、SDK、工具函数；严格控制不引入运行时耦合。

### 3.2 后端目录规范

```text
apps/api/src/modules/<domain>
```

- `controller.ts`：负责入参、出参、鉴权依赖和调用 service。
- `service.ts`：负责业务规则、事务边界和跨聚合协调。
- `repo.ts`：负责数据访问，必须统一注入当前用户条件。
- `dto.ts`：负责入参 DTO 和校验规则。

```text
apps/api/src/common
```

- `auth`：鉴权、用户上下文、权限守卫。
- `user-context`：当前用户上下文注入、当前用户数据隔离拦截或工具。
- `errors`：统一错误码、异常类和异常过滤器。
- `logging`：logger、traceId、中间件。

### 3.3 小程序目录规范

```text
apps/miniapp/src
```

- `pages`：页面目录，按业务页面划分。
- `components`：可复用组件，组件文件使用 PascalCase。
- `services`：接口请求封装和业务 API。
- `hooks`：React hooks，文件名使用 `useXxx.ts`。
- `stores`：全局或模块状态。
- `styles`：主题变量、全局样式、组件样式。
- `utils`：纯工具函数。

## 4. 编码规范

### 4.1 命名

- 变量/函数：camelCase。
- 类型/接口/类/组件：PascalCase。
- 常量：UPPER_SNAKE_CASE。
- Taro 页面目录：使用小写或 kebab-case，保持全项目一致。
- 组件文件：PascalCase。
- hook 文件：`useXxx.ts`。
- 数据库字段：snake_case。

### 4.2 导入顺序

1. 外部依赖。
2. 内部模块。
3. 类型导入，使用 `import type`。
4. 样式文件。

### 4.3 TypeScript 强约束

- 必须启用 `strict: true`。
- 禁止无理由 `any`；需要动态类型时使用 `unknown` + 类型收窄/校验。
- 禁止无校验断言；`as Xxx` 必须有 schema、guard 或外部契约来源。

### 4.4 编码与换行规范

- 文件编码：UTF-8 无 BOM。
- 换行符：CRLF。
- 必须启用 EditorConfig，禁止编辑器忽略 `.editorconfig`。
- 保留文件末尾换行，禁止行尾空白。

### 4.5 格式化规范

- 使用 Prettier 统一格式化。
- 使用 ESLint 做规范与质量检查。
- 保存或提交前自动格式化，避免手工格式差异。

### 4.6 校验与约束

- 提交前或 CI 中必须检查编码、行尾、格式化、类型和构建。
- 违反上述规范不允许合并或发布。

## 5. 接口契约与响应规范

### 5.1 统一响应 Envelope

成功：

```json
{ "code": 0, "message": "ok", "data": {}, "traceId": "..." }
```

失败：

```json
{ "code": 10001, "message": "VALIDATION_ERROR", "details": {}, "traceId": "..." }
```

- `traceId`：每次请求必须存在，由前端透传或后端生成。
- `code`：业务错误码，不直接用 HTTP 状态码替代。
- HTTP 状态码仍要正确使用：400、401、403、404、409、429、500。

### 5.2 分页统一

```json
{
  "code": 0,
  "message": "ok",
  "data": { "pageNo": 1, "pageSize": 10, "total": 123, "list": [] },
  "traceId": "..."
}
```

### 5.3 契约来源

- OpenAPI 3.0 作为接口单一事实来源。
- 前端类型与客户端由 OpenAPI 生成或集中维护，避免散落字符串与重复类型。
- 接口字段、错误码、分页结构变更时，必须先更新 OpenAPI，再更新实现。

## 6. 安全编码

### 6.1 微信登录与身份识别

- 一期使用微信小程序登录，不使用手机号密码登录。
- 小程序端获取 `wxCode` 后提交给后端。
- 后端使用 `wxCode` 调微信接口静默换取 `openid`。
- 用户最终身份以 `users.openid` 识别；同一个 `openid` 只能对应一个用户。
- 后端根据 `openid` 创建或返回用户，并签发 JWT。
- 小程序端将 JWT 存储于 `Taro.setStorageSync('token', ...)`，请求时携带 `Authorization: Bearer <token>`。
- 后端业务接口不得信任前端传入的 `userId`，必须从 JWT 和服务端用户上下文中取得当前用户。

### 6.2 输入验证

- 后端统一使用 schema 校验，Zod、class-validator、Joi 任选其一，但全项目必须统一。
- 前端可做表单级校验，但不得替代后端校验。
- 金额、日期、分类、账户、标签等关键字段必须在后端二次校验。

### 6.3 当前用户数据隔离

- 所有业务表查询和写入必须带当前 `userId` 条件。
- 账单、账户、分类、标签等资源只能被所属用户读取和操作。
- 跨用户数据访问必须返回 403 或等价业务错误。
- 删除、编辑、设默认等写操作必须先校验资源归属。

### 6.4 凭证与敏感信息

- JWT secret、微信 app secret、大模型 key、服务器私钥不得提交到 Git。
- 日志不得明文输出 token、secret、私钥、手机号等敏感信息。
- 配置文件必须区分本地、测试和生产环境。

## 7. 错误处理与错误码

### 7.1 错误分类

- ValidationError：参数错误，HTTP 400。
- UnauthorizedError：未登录，HTTP 401。
- ForbiddenError：无权限或跨用户访问，HTTP 403。
- NotFoundError：资源不存在，HTTP 404。
- ConflictError：并发或唯一约束冲突，HTTP 409。
- RateLimitError：限流，HTTP 429。
- InternalError：未预期错误，HTTP 500。

### 7.2 规则

- 业务错误必须可预期，使用自定义错误类和错误码。
- 不可预期错误必须记录完整日志，包含 `traceId`，对外返回统一 message，不泄露堆栈。
- 禁止在业务分支里滥用 `console.error`；可预期错误用 warn/info，未预期错误用 error。

## 8. 后端实现规范

### 8.1 幂等

- 写接口必须考虑重复提交和重试。
- 优先使用业务唯一键保证幂等。
- 必要时使用幂等 token 或去重表。

### 8.2 事务边界

- 一个 service 方法作为一个事务边界，按业务需要开启事务。
- 账单保存、批量保存、账单编辑、账单删除和账户余额变更必须在同一事务内完成。
- 事件或消息场景优先使用 Outbox，谨慎使用分布式事务。

### 8.3 返回数据

- 后端接口返回数据时，优先通过联表查询或明确聚合组装一次性返回前端所需字段。
- 管理类详情、审计相关接口和需要排障的接口必须返回创建时间和更新时间。
- 列表接口只返回页面展示和交互必需字段，避免过度返回。

## 9. 小程序实现规范

### 9.1 认证与登录

- 小程序启动后优先检查本地 token。
- token 不存在或失效时，调用 `Taro.login` 获取 `wxCode`。
- 后端收到 `wxCode` 后换取微信 `openid`，并以 `users.openid` 创建或识别用户。
- 登录成功后保存 token，并拉取当前用户基础信息。

### 9.2 API 请求规范

- 封装统一请求工具 `services/request.ts`。
- 请求基于 `Taro.request`，自动注入 token 和 traceId。
- 自动处理 Envelope 响应。
- 统一处理 401、403、参数错误和网络错误。
- 复用后端同一套 REST 接口，不为小程序单独新建接口；确需差异化时在现有接口入参中扩展。

### 9.3 UI 实现规范

- 页面和组件必须遵循 `AI文档/UI规范.md`。
- 颜色、字号、圆角、间距、按钮、弹窗、抽屉、Toast 不允许页面单独发明样式。
- 分类图标使用 Unicode Emoji，来自分类图标库。
- 操作类图标固定在前端组件中，不进入分类图标库。

### 9.4 分包规范

- 主包 `pages/` 体积不超过 1.5MB，超出必须拆分为分包。
- 低频功能如报表、设置、管理页可优先下沉分包。
- 分包配置在 `app.config.ts` 中统一声明。

## 10. 日志与监控

### 10.1 日志字段规范

日志至少包含：

- `traceId`
- `userId`
- `path`
- `method`
- `status`
- `costMs`

PII 脱敏：手机号、token、密码、密钥、私钥严禁明文输出。

### 10.2 操作日志

一期优先记录以下关键操作：

- 登录成功或失败。
- 新增、编辑、删除账单。
- 新增、编辑、删除分类。
- 新增、编辑、删除账户。
- 新增、编辑、删除标签。
- 语音解析成功或失败。
- 跨用户数据访问拦截。

## 11. 测试规范

### 11.1 后端测试

- 单元测试覆盖核心 service。
- 必须覆盖当前用户数据隔离逻辑。
- 必须覆盖账户余额变更、账单编辑、账单删除、分类删除限制、账户删除限制。
- 集成测试可使用 testcontainers 或 docker-compose。

### 11.2 小程序测试

- 覆盖登录、记录页、手工记账、账户选择、日期选择、标签选择、语音确认保存等关键链路。
- 关键页面需按 `UI规范.md` 做视觉走查。

### 11.3 E2E 最小链路

- 登录。
- 新增一笔账单。
- 查看账单列表。
- 编辑账单。
- 删除账单。

## 12. CI/CD 质量门禁

- lint 必须通过。
- typecheck 必须通过。
- test 必须通过。
- 构建产物必须可复现，lockfile 必须提交。
- 依赖安全扫描建议开启。
- 每次提交线上仓库前都需要进行一次构建校验。
